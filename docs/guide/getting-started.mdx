# Getting Started

There are two flows for running a Lynx project against Tamer. Pick one — your selection sticks for the rest of the page.

<Tabs queryKey="flow">
<Tab label="Tamer App (prebuilt)" value="tamer-app">

The fastest path. Install the **prebuilt Tamer Dev App** on your phone or simulator, run `t4l start` from your project, scan the QR. No Xcode / Android SDK / native build required.

:::tip Use this flow when…
You want to iterate on Lynx UI without touching native code, or you don't have Xcode / Android SDK installed locally.
:::

## 1. Install the prebuilt app

Download from the [latest release](https://github.com/tamer4lynx/tamer4lynx/releases/latest):

<Tabs queryKey="platform">
  <Tab label="iOS" value="ios">
    **Simulator** — download `TamerDevApp-Simulator-<version>.zip`, unzip, install on a booted simulator:

    ```bash
    unzip TamerDevApp-Simulator-<version>.zip
    xcrun simctl install booted ./TamerDevApp.app
    xcrun simctl launch booted com.nanofuxion.tamerdevapp
    ```

    **Device** — download `TamerDevApp-Device-Unsigned-<version>.zip`. The `.app` ships unsigned to keep the release portable. Re-sign with your own Apple identity, embed a provisioning profile that matches `com.nanofuxion.tamerdevapp` (or rewrite the bundle id), wrap as IPA, install. Step-by-step in the bundled `SIGN-AND-INSTALL.txt`.
  </Tab>
  <Tab label="Android" value="android">
    Download `TamerDevApp-<version>.apk`. Universal APK — supports `arm64-v8a`, `armeabi-v7a`, `x86_64` (emulators + physical devices).

    ```bash
    adb install TamerDevApp-<version>.apk
    adb shell am start -n com.nanofuxion.tamerdevapp/.MainActivity
    ```
  </Tab>
</Tabs>

## 2. Install the CLI in your project's package manager

You still need `t4l start` to run the dev server. Install the CLI globally:

<PackageManagerTabs
  command={{
    npm: 'npm i -g @tamer4lynx/cli@latest',
    yarn: 'yarn global add @tamer4lynx/cli@latest',
    pnpm: 'pnpm add -g @tamer4lynx/cli@latest',
    bun: 'bun add -g @tamer4lynx/cli@latest',
  }}
/>

## 3. Run the dev server

From your Lynx project root:

```bash
t4l start
```

The dashboard prints a QR. Scan it inside the prebuilt Tamer Dev App — bundles hot-reload over the dev server. Done.

## When this flow runs out

Switch to the **Dev Client** flow if you need any of:

- A **custom native module** or **custom element** that isn't shipped in the prebuilt app.
- A different **bundle id**, app icon, or signing identity (e.g. shipping to TestFlight under your own team).
- A **smaller binary** that includes only the Tamer packages your app actually uses.

</Tab>
<Tab label="Dev Client (build it yourself)" value="dev-client">

Build your own dev host from your project. Picks up your custom native modules, your bundle id, your signing identity. Required if the prebuilt Tamer App doesn't carry the Tamer packages your app needs.

:::tip Use this flow when…
You're authoring or consuming custom native modules / elements, need control over signing, or want a single binary scoped to your app's actual dependency set. New to native module authoring? See [Native Modules & Elements](/guide/native-modules).
:::

## Prerequisites

Node.js **20.19+** or **Bun**; **Android SDK** for Android builds; **Xcode** on macOS for iOS.

## 2. Create a Lynx project (or use existing)

Create a new Lynx project with Rspeedy:

<PackageManagerTabs
  command={{
    npm: 'npm create rspeedy',
    yarn: 'yarn create rspeedy',
    pnpm: 'pnpm create rspeedy',
    bun: 'bun create rspeedy',
  }}
/>

Then `cd` into the project directory you just created and install dependencies. Rspeedy's default name is **`my-app`**, but the create wizard lets you pick anything — substitute your actual directory name below:

<PackageManagerTabs command="install" />

**Using a different Lynx binding?** The `create` step differs per framework:

- **[miso-lynx](https://lynxjs.haskell-miso.org/)** — Haskell-on-Lynx. Project scaffold and directory layout follow the miso-lynx docs.
- **[VueLynx](https://vue.lynxjs.org/)** — Vue 3 on Lynx via `@lynx-js/vue`. Use the VueLynx starter; the produced directory name will match its template.
- **Other bindings** — anything that emits a Lynx bundle works. Skip the `create rspeedy` step, `cd` into your existing project, then run `t4l init`.

After install, run `t4l init` from the project root to wire up Tamer.

## 3. Project layout: single folder vs monorepo

A Tamer4Lynx project can be **one folder** or a **monorepo**:

| Layout | Structure | When to use |
|--------|-----------|-------------|
| **Single folder** | `tamer.config.json`, Lynx app (`src/`, `lynx.config.ts`), `android/`, `ios/` all in the same directory | New projects, simple apps. Run `t4l init` and omit Lynx project path (or use `.`). |
| **Monorepo** | `tamer.config.json` at root; Lynx app in `packages/myapp`; `android/`, `ios/` at root | Multiple packages, shared tooling. Set `lynxProject: "packages/myapp"` in config. |

The example in this repo uses a monorepo layout. For a single-folder project, copy a Lynx app into a new directory, run `t4l init` there, and create native projects with `t4l create android` / `t4l create ios`.

## 4. Initialize config

From the project root (where `tamer.config.json` will live):

```bash
t4l init
```

Or run `t4l` with no arguments for interactive setup. The wizard prompts for Android app name, package name, SDK path, iOS bundle ID, and Lynx project path (leave blank for single-folder layout).

**TypeScript / `tsconfig.json`:** On save, `t4l init` updates the Lynx app's `tsconfig.json` when it exists (root, or under `lynxProject` if you set one): it may **flatten project references** to avoid TS6310 when building, then generates **`.tamer/tamer-components.d.ts`** and ensures **`include`** for it (see [Ambient types](/packages/tooling/tamer-ambient-types)).

## 5. Add Tamer packages

Dev launcher stack (dev-app, dev-client, and their `@tamer4lynx` dependencies):

```bash
t4l add-dev
```

Or, if you do not need the dev client / dev app — core only (app-shell, screen, router, insets, transports, system-ui, icons):

```bash
t4l add-core
```

Then link native projects (required after adding packages unless [autolink](/guide/configuration) is enabled):

<Tabs queryKey="platform">
  <Tab label="Android" value="android">
    ```bash
    t4l link android
    ```
  </Tab>
  <Tab label="iOS" value="ios">
    ```bash
    t4l link ios
    ```
  </Tab>
</Tabs>

## 6. Build the dev client

| Flag | Short | Description |
|------|-------|-------------|
| `--debug` | `-d` | Debug build with dev client (default). Requires `tamer-dev-client` for QR/HMR. |
| `--release` | `-r` | Release build **without** dev client; **unsigned**. |
| `--production` | `-p` | **Signed** build for App Store / signed distribution. Requires `t4l signing` first. |
| `--install` | `-i` | After build: Android installs APK; iOS **debug** installs to **simulator**; **production** (`-p`) installs to a **physical device only** (`devicectl`). |
| `--embeddable` | `-e` | Output embeddable artifacts for existing apps; use **`t4l build android --embeddable`** (Android only). |

**iOS quick reference:** `-p` always targets **devices** (`iphoneos`). `-d -i` targets the **simulator** and installs there. **`t4l build ios -p --ipa`** exports an IPA after archiving.

**Platform:** `t4l build` requires **`android` or `ios`** each time (one platform per command).

<Tabs queryKey="platform">
  <Tab label="Android" value="android">
    ```bash
    t4l build android -d               # debug Android (dev client if tamer-dev-client installed)
    t4l build android --install        # debug, install to running emulator/device
    t4l build android --release        # release Android, no dev client, unsigned
    t4l build android --embeddable     # embeddable output (release-style)
    ```
  </Tab>
  <Tab label="iOS" value="ios">
    ```bash
    t4l build ios --install            # debug iOS, install to booted simulator
    t4l build ios -p                   # signed production (device); use after signing setup
    t4l build ios -p --ipa             # archive + export IPA
    ```
  </Tab>
</Tabs>

## 7. Run the dev server

```bash
t4l start
```

The terminal prints a QR. Scan it from the dev client you just built (or any other dev-client-equipped binary). Bundles hot-reload over the dev server.

## Dev host and native modules

The **debug / HMR** native app you build is **made from your project** after **`t4l link`**, so the iOS and Android binaries include **the native modules your app actually uses** (autolinked pods, Gradle modules, registration).

Whenever you add or remove **`@tamer4lynx/*`** native packages, run **`t4l link`** (or **`t4l link ios`** / **`t4l link android`**) so autolinking updates native projects — unless **`autolink: true`** in `tamer.config.json` runs `t4l link` after installs (see `t4l autolink-toggle`).

## CLI and workspace notes

- **Environment files:** On startup, `t4l` loads **`.env`** then **`.env.local`** beside `tamer.config.json` (Android signing passwords, App Store Connect API keys for **`t4l build ios -p --ipa`**, and similar). See [Commands](/reference/commands) (Global section).
- **Interactive flows:** **`t4l init`** and **`t4l signing`** are Ink wizards. **`t4l start`** is an Ink dashboard: **`r`** rebuild bundle, **`l`** toggle logs, **`q`** quit.
- **TypeScript:** With **`syncTamerComponentTypes`** enabled (default in new configs), **`t4l init`** / **`t4l link`** can adjust the Lynx app `tsconfig.json` and generate **`.tamer/tamer-components.d.ts`**. Details: [Ambient types](/packages/tooling/tamer-ambient-types).

## Framework agnostic

Tamer4Lynx is conceptually Lynx framework agnostic. Native modules and tooling work with any Lynx binding, including [miso-lynx](https://lynxjs.haskell-miso.org/) and [VueLynx](https://vue.lynxjs.org/) (`@lynx-js/vue`). Some packages like **tamer-router** are specifically designed for `@lynx-js/react`.
